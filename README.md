# KaijuDrop (КайдзюДроп) — Премиум Аниме-Мерч

КайдзюДроп — это современный интернет-магазин премиального аниме-мерча, фигурок, уличной одежды и аксессуаров. Проект разработан на базе Nuxt 4 с использованием Drizzle ORM, SQLite (`better-sqlite3`), TailwindCSS и Pinia.

---

## 🧪 Тестирование проекта (Задание 4)

В проекте настроено комплексное автоматизированное тестирование с использованием современных инструментов (**Vitest** и **@nuxt/test-utils**). Мы объединили два уровня тестирования для обеспечения максимальной надежности ключевого функционала:

### 1. Модульное тестирование (Unit Testing)
* **Цель**: Проверка чистой бизнес-логики без привязки к серверам или базам данных.
* **Объект тестирования**: Стейт-менеджер корзины товаров (**Pinia Store** в `stores/cart.ts`).
* **Покрытые сценарии**:
  * Добавление товаров в корзину (включая группировку одинаковых вариантов).
  * Изменение количества товаров в корзине.
  * Удаление товаров из корзины.
  * Расчет суммарной стоимости заказа с учетом количества и цен.
  * Очистка корзины.
* **Расположение файлов**: `tests/unit/cart.test.ts`

### 2. Интеграционное тестирование API (Integration API Testing)
* **Цель**: Тестирование взаимодействия сервера, бизнес-логики и базы данных.
* **Объект тестирования**: API-эндпоинты каталога и оформления заказов.
* **Покрытые сценарии**:
  * **Каталог товаров (`/api/merch`)**:
    * Получение списка товаров с вложенными вариантами и актуальным остатком на складе.
    * Фильтрация товаров по категориям (`category`) и франшизам (`franchise`).
    * Детальное получение товара по ID (включая обработку ошибки 404).
  * **Оформление заказа (`/api/checkout`)**:
    * Успешная покупка: создание заказа, сохранение позиций в БД и корректное списание остатков со склада.
    * Валидация входных данных (ошибка при пустой корзине).
    * Контроль транзакционности: откат изменений (rollback) при попытке купить товар, которого нет в наличии.
* **Расположение файлов**: `tests/api/merch.test.ts`, `tests/api/checkout.test.ts`

## 🛠️ Особенности архитектуры тестов

1. **Изоляция тестовой базы данных**:
   * Для тестов автоматически создается и используется изолированный файл БД `sqlite_test.db` (определяется динамически по переменным окружения `VITEST` и `NODE_ENV`).
   * Перед запуском каждого интеграционного теста база данных полностью очищается и заново заполняется стабильным набором тестовых данных (сид-данными). Это гарантирует независимость тестов друг от друга.
2. **Изоляция состояния (Pinia)**:
   * В файле настроек `tests/setup.ts` перед каждым тестом пересоздается инстанс Pinia, предотвращая утечки состояния между тестами.
3. **Безопасность транзакций**:
   * Проверено поведение транзакций базы данных при многопоточной работе SQLite с использованием синхронных транзакций `better-sqlite3`.

## 🚀 Запуск тестов

Убедитесь, что установлены все зависимости:

```bash
bun install
```

### Запуск всех тестов
Запустите команду для однократного прогона всех тестов в консоли:

```bash
bun run test
# или напрямую через vitest:
bunx vitest run --run
```

### Запуск в режиме наблюдения (Watch Mode)
Для разработки с автоматическим перезапуском тестов при изменении кода:

```bash
bunx vitest
```
